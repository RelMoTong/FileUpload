# 磁盘爆满紧急清理功能修复说明

## 📅 修复日期
2026年1月9日

## 🐛 问题描述

用户反映磁盘空间达到100%后，程序提示"自动清理未找到可删除的文件"，导致无法继续上传。

### 截图证据

从用户提供的截图可以看到：
- **右下角警告弹窗**：磁盘空间不足 - 目标: 4% | 备份: 100% | 阈值: 5%
- **日志中反复出现**：
  - ⚠️ 磁盘空间达到阈值，触发自动清理
  - ⚠️ 自动清理未找到可删除的文件
  - ⚠️ [DISK_LOW] 磁盘空间仍不足 | target=4% backup=n/a threshold=5%

### 问题表现

1. 备份磁盘已经100%满
2. 目标磁盘只剩4%空间（低于5%阈值）
3. 程序检测到空间不足，触发自动清理
4. **但是找不到可删除的文件**
5. 程序无法继续上传，陷入死循环

## 🔍 根本原因分析

### 代码问题定位

在 `_auto_cleanup_once()` 方法中（原第1231-1248行）：

```python
cutoff_time = 0
if self.auto_delete_keep_days > 0:
    cutoff_time = time.time() - (self.auto_delete_keep_days * 24 * 3600)

# ... 扫描文件 ...

file_time = stat.st_ctime if os.name == 'nt' else stat.st_mtime
if cutoff_time and file_time > cutoff_time:
    continue  # ❌ 跳过不满足删除条件的文件
```

### 问题分析

**保留天数限制过于严格**：

1. 默认设置 `auto_delete_keep_days = 10`（保留10天）
2. 只删除**10天前**创建的文件
3. 即使磁盘100%满，也**不会删除10天内的文件**
4. 如果清理目录中所有文件都是最近10天创建的，就**找不到可删除的文件**

### 为什么会这样设计？

这是一个**数据安全机制**，防止误删最近的重要文件。

**设计初衷**：
- 保护最近的文件不被误删
- 只清理"旧文件"
- 适用于正常运营场景

**问题场景**：
- 磁盘突然爆满（如大量文件涌入）
- 所有文件都很新（如刚备份的）
- 保留天数限制成为瓶颈

### 真实场景举例

假设用户的使用场景：
1. 每天备份大量图片到备份盘
2. 最近7天产生了大量数据
3. 备份盘已满（100%）
4. 但所有文件都是7天内的
5. 程序无法删除任何文件（因为设置保留10天）
6. **陷入死循环：空间不足 → 触发清理 → 找不到文件 → 仍然不足 → 重复...**

## ✅ 修复方案

### 解决思路

引入**两级清理模式**：

1. **普通模式**（磁盘使用 5%-80%）
   - 尊重保留天数设置
   - 只删除超过保留天数的文件
   - 保护最近的重要文件

2. **紧急模式**（磁盘使用 >95% 或 剩余<5%）
   - **忽略保留天数限制**
   - 直接清理最旧的文件
   - 优先保证程序能继续运行

### 修复1: 增加紧急模式参数

修改 `_auto_cleanup_once()` 方法：

```python
def _auto_cleanup_once(self, emergency_mode: bool = False) -> None:
    """自动清理文件
    
    Args:
        emergency_mode: 紧急模式，磁盘严重不足时忽略保留天数限制
    """
    # ...
    
    # 紧急模式：磁盘严重不足时忽略保留天数限制
    cutoff_time = 0
    if self.auto_delete_keep_days > 0 and not emergency_mode:
        cutoff_time = time.time() - (self.auto_delete_keep_days * 24 * 3600)
        self.log.emit(f"🧹 自动清理模式: 普通（保留{self.auto_delete_keep_days}天内文件）")
    elif emergency_mode:
        self.log.emit("🚨 自动清理模式: 紧急（忽略保留天数，直接清理最旧文件）")
    
    # ... 扫描和删除逻辑 ...
```

**改进点**：
- 新增 `emergency_mode` 参数
- 紧急模式下 `cutoff_time = 0`，不过滤任何文件
- 清晰的日志提示当前清理模式

### 修复2: 智能判断是否启用紧急模式

修改 `_ensure_disk_space()` 方法：

```python
def _ensure_disk_space(self) -> bool:
    # 获取磁盘空间
    tf_ok, _, _ = self._disk_ok(self.target)  # 目标盘剩余百分比
    bf_ok, _, _ = self._disk_ok(self.backup)   # 备份盘剩余百分比
    
    # 检查是否需要紧急清理（磁盘严重不足：剩余<5% 或 使用>95%）
    emergency_mode = (tf_ok < 5.0) or (backup_check and bf_ok < 5.0)
    
    if should_cleanup:
        if emergency_mode:
            self.log.emit("🚨 磁盘空间严重不足（<5%），触发紧急清理模式")
        else:
            self.log.emit("⚠️ 磁盘空间达到阈值，触发自动清理")
        
        # 第一次尝试：按当前模式清理
        self._auto_cleanup_once(emergency_mode=emergency_mode)
        
        # 重新检查磁盘空间
        # ...
        
        # 如果清理后仍然严重不足，且之前不是紧急模式，则再次尝试紧急清理
        still_critical = (tf_ok < 5.0) or (backup_check and bf_ok < 5.0)
        if still_critical and not emergency_mode:
            self.log.emit("🚨 普通清理后空间仍不足，启用紧急清理模式（忽略保留天数）")
            self._auto_cleanup_once(emergency_mode=True)
```

**改进点**：
- 自动判断是否需要紧急模式（剩余<5%）
- 先尝试普通清理，不够再尝试紧急清理
- 两次清理机会，提高成功率

### 修复3: 优化日志提示

## ⚠️ 注意事项

### 1. 紧急模式触发条件

- 剩余空间 **< 5%**
- 或者使用空间 **> 95%**
- 任一磁盘（目标盘或备份盘）满足条件即触发

### 2. 文件删除顺序

无论哪种模式，都按**文件创建时间从旧到新**删除：
- 最旧的文件优先删除
- 删除到空间充足为止
- 不会一次性全部删除

### 3. 安全保护

- 只删除清理目录中的文件
- 删除前检查文件时间戳
- 异常文件跳过，不影响整体清理
- 详细的日志记录每次清理结果

### 4. 用户配置建议

**保留天数设置建议**：
- 小容量磁盘（<500GB）：建议3-5天
- 中容量磁盘（500GB-2TB）：建议7-10天
- 大容量磁盘（>2TB）：建议10-30天

**清理目录设置**：
- 必须设置清理目录
- 建议设置为备份目录或归档目录
- 不要设置为源文件目录（防止误删源文件）

## 🔧 技术细节

### 关键变量

```python
emergency_mode: bool           # 是否紧急模式
cutoff_time: float            # 截止时间戳（普通模式使用）
auto_delete_keep_days: int    # 保留天数配置
```

### 判断逻辑

```python
# 紧急模式判断
emergency_mode = (tf_ok < 5.0) or (backup_check and bf_ok < 5.0)

# 保留天数处理
if self.auto_delete_keep_days > 0 and not emergency_mode:
    cutoff_time = time.time() - (self.auto_delete_keep_days * 24 * 3600)
elif emergency_mode:
    cutoff_time = 0  # 不过滤，全部可删
```

### 清理流程

```
1. 检查磁盘空间
2. 判断清理模式（普通/紧急）
3. 扫描清理目录
4. 过滤文件（紧急模式跳过过滤）
5. 按时间排序
6. 从最旧开始删除
7. 删除到空间充足为止
8. 记录清理结果
```
